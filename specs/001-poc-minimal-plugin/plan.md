# Implementation Plan: PoC & Minimal Kotlin/Native Plugin for Pico

**Branch**: `001-poc-minimal-plugin` | **Date**: 2026-07-02 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/001-poc-minimal-plugin/spec.md`

## Summary

Phase 0/1 from `ROADMAP.md`: prove that Kotlin/Native code can run on a
Raspberry Pi Pico (RP2040) via cinterop with the Pico SDK and UF2
generation, then deliver a minimal Gradle plugin `com.anjo.kopico`
automating this experience (extension DSL `board`/`sdkPath`,
auto-registration of a custom Kotlin/Native target, auto-cinterop,
auto-provisioning of the entire toolchain — Pico SDK, ARM GCC, `picotool`,
OpenOCD — with local cache and version pinning). Key technical risk:
Kotlin/Native does not officially support bare-metal ARM Cortex-M as a
target — Phase 0 is precisely a spike verifying whether this can be worked
around (see `research.md`).

## Technical Context

**Language/Version**: Kotlin 2.4.0 (plugin: `kotlin("jvm")` on Gradle
9.5.1/JVM; generated target code: Kotlin/Native custom target, without KMP)

**Primary Dependencies**: `java-gradle-plugin`, `io.github.oshai:kotlin-logging`,
Kotest (`kotest-runner-junit5`, `kotest-assertions-core`), Gradle TestKit
(plugin functional tests)

**Storage**: N/A (no database involved) — local tool cache on disk under
`<gradleUserHome>/caches/kopico/<tool>/<version>/`

**Testing**: Kotest (`FunSpec`/`BehaviorSpec`) for unit logic + Gradle
TestKit for plugin functional tests (applying the plugin to a test project,
running a build)

**Target Platform**: The plugin runs on the JVM (Gradle daemon, Linux);
code generated by the plugin targets bare-metal ARM (RP2040
`thumbv6m-none-eabi`, RP2350 `thumbv8m.main-none-eabi`/`eabihf`) without an
operating system

**Project Type**: Gradle plugin (library) — a single Gradle module, plus a
sample `examples/blink` project consuming the plugin

**Performance Goals**: SC-002 — full setup from scratch to a working UF2 in
< 15 minutes (including the first tool download); otherwise the build is
not performance-sensitive at this stage

**Constraints**: Gradle 9.5.1 and Kotlin 2.4.0 exactly (Principle I); no
KMP (Principle I); dev/CI environment limited to Linux x86_64 (Assumptions
in spec.md); auto-provisioning must work offline after the first download
(SC-006, FR-014)

**Scale/Scope**: 4 board variants (pico, pico_w, pico2, pico2_w), 3 Pico
SDK libraries in scope for this phase (`pico_stdlib`, `hardware_gpio`,
`hardware_pwm`) + `pico_cyw43_arch` for `_w` variants, one sample blink
project

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Assessment | Rationale |
|---|---|---|
| I. Strict toolchain version control | PASS | The plan keeps Gradle 9.5.1 / Kotlin 2.4.0 exactly; custom Kotlin/Native target without `kotlin("multiplatform")` (see research.md for the mechanism) |
| II. 100% Kotlin, zero Java | PASS | All plugin code (including the UF2 writer — see research.md decision) is in Kotlin; logging exclusively via `kotlin-logging` |
| III. Test-First in Kotest | PASS | Unit and functional tests (TestKit) in Kotest; no direct JUnit annotations |
| IV. Gradle Plugin Best Practices | PASS | `java-gradle-plugin`, extension built on `Provider`/`Property`, tasks separated from the extension |
| V. Quality and static analysis | PASS (procedural) | ktlint/detekt as CI gates; `ponytail` used for implementation and verification; commits via `git-committer` — enforced during the execution phase, not the architecture |

No violations — `Complexity Tracking` remains empty.

**Re-check after Phase 1 (design)**: `research.md`, `data-model.md`, and
`contracts/extension-dsl.md` introduce no new violations. Calls to external
tools (`konanc`, `cinterop`, `arm-none-eabi-gcc`, `picotool`, `openocd`,
`ld.lld`) via Gradle `Exec` do not break Principle II — this is
orchestration of external processes, not plugin source code. PASS, no
changes.

**Re-check after Phase 0 (PoC, 2026-07-03)**: the pipeline verified on
physical hardware forced design corrections (see `poc/RESULTS.md`,
`poc/konan-target-spike.md`): (1) added `KotlinNativeProvisioner` with a
`.bc` attribute patch; (2) added `LinkTask` — `konanc -produce static`
only produces a library, the final ELF is produced by linking against
pico-sdk with a C shim and lld; (3) the Kotlin `Uf2Writer` was replaced
with a call to the provisioned `picotool uf2 convert` (a tool already
required by FR-013 anyway — reuse instead of reimplementation, in line
with the ponytail principles; the C shim/wrapper/linker-script resources
are files injected into the user's build, not plugin code — Principle II
not violated). PASS.

### Source Code (repository root)

```text
src/main/kotlin/com/anjo/kopico/
├── KopicoPlugin.kt              # java-gradle-plugin entry point
├── KopicoExtension.kt           # public DSL: board, sdkPath (Provider/Property)
├── BoardVariant.kt              # enum/sealed: pico, pico_w, pico2, pico2_w + triple/CYW43 info
├── provisioning/
│   ├── ToolCache.kt             # <gradleUserHome>/caches/kopico/<tool>/<version>/
│   ├── PicoSdkProvisioner.kt    # clone/cache Pico SDK (FR-013/FR-014)
│   ├── ArmToolchainProvisioner.kt
│   ├── PicotoolProvisioner.kt
│   ├── OpenOcdProvisioner.kt
│   └── KotlinNativeProvisioner.kt # K/N 2.4.0 distribution + .bc attribute patch (poc § Round 3)
└── tasks/
    ├── CinteropTask.kt          # `cinterop` with retargeting overrides
    ├── CompileNativeTask.kt     # `konanc` with retargeting → libapp.a (produce static)
    ├── LinkTask.kt              # final link against pico-sdk: C shim + wrapper + lld + linker script
    └── GenerateUf2Task.kt       # provisioned `picotool uf2 convert` on the ELF from LinkTask

src/main/resources/kopico/       # resources injected into the user's build
├── kopico_shim.c                # pthread/mmap/TLS/cond_var stubs (poc/blink/kopico_shim.c)
├── kopico_stdio_globals.c
├── wrapper.c                    # GPIO/CYW43 bridge + call into Kotlin main
└── memmap_kopico.ld             # linker script: .got in FLASH

src/test/kotlin/com/anjo/kopico/
├── BoardVariantTest.kt          # Kotest FunSpec
├── KopicoExtensionTest.kt
├── provisioning/ProvisionersTest.kt
├── tasks/CinteropTaskTest.kt
└── KopicoPluginFunctionalTest.kt # Gradle TestKit

examples/blink/
├── build.gradle.kts             # applies the plugin, board = "pico"
└── src/nativeMain/kotlin/Main.kt # sample blink code (gpio_put)
```

**Structure Decision**: A single Gradle plugin module (no multi-module
setup) — in line with Principle IV (`java-gradle-plugin` + convention is
most simply realized as a single project at this stage).
`examples/blink` is a separate, independent Gradle project consuming the
plugin via Maven Local (FR-009), serving as the end-to-end acceptance test
for User Story 1/2.
